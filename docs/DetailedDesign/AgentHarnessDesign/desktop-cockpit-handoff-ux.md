# Desktop Cockpit + Approval/Handoff 信息架构详细设计

## 1. 目标与边界

本设计定义 Enterprise Agent Harness 的 Desktop Cockpit、审批提示、通知与人工接管信息架构。它承接原始需求中的对话任务、内网浏览器操作、文件整理、定时任务、截图归档、知识引用、用户确认、高风险操作拦截、人工接管和审计留痕要求，并落实 RALPLAN 选定的 **Desktop Client + Local Agent Runtime/Daemon + Enterprise Control Plane + Isolated Workers + Unified Audit/Event Spine** 架构。

Desktop Cockpit 是用户座舱，不是主执行引擎。它负责让用户看见 Agent 正在做什么、准备做什么、为什么需要确认、何时需要接管，以及已产生哪些结果、引用和审计摘要。所有执行事实来自 Local Runtime/Daemon 的 Run、事件、审批、产物和通知快照。

### 1.1 设计目标

- 让用户通过一个清晰的桌面座舱发起任务、观察进度、控制 Run、处理审批、接收通知和查看执行记录。
- 让审批与接管成为可恢复、可追踪、可审计的任务状态，而不是一次性 UI 弹窗。
- 将高风险动作的目标、范围、影响、风险等级、策略来源、可撤销性和证据材料以可理解方式展示给用户。
- 支持 UI 关闭、重启或后台运行后的恢复：重新打开 Desktop 后能恢复任务卡、待审批、通知历史、接管状态和产物摘要。
- 为 Runtime、Policy/Approval、Browser/File Worker、Knowledge/Citation、Audit Query 等详细设计提供稳定的用户可见语义。

### 1.2 明确边界

本文件只描述信息架构与交互语义，不包含：

- 数据库表结构、持久化字段或索引设计。
- 端点级 API、IPC channel 名称或序列化字段。
- 具体 React 组件、路由、状态管理、Electron 实现或视觉高保真原型。
- 具体浏览器自动化、文件处理、模型调用或调度实现。
- 实施排期、工作量估算、SaaS 多租户扩展或代码 POC。
- Policy/Approval Engine、Browser Worker、File Worker、Knowledge/Memory 或 Audit Query 的完整内部设计；本文件只定义它们在 Desktop Cockpit 中的可见入口与信息结构。

## 3. 职责划分

### 3.1 模块职责

| 模块 | 在 Desktop Cockpit 中的职责 | 不负责 |
| --- | --- | --- |
| Desktop Cockpit Shell | 提供导航、任务入口、通知入口、设置入口、全局状态提示和恢复提示 | 执行工具动作、保存长期执行事实、绕过 Runtime 更改任务状态 |
| Conversation / Task Workspace | 接收用户目标、展示对话、任务拆解、结果、引用、产物和可执行建议 | 直接调用真实工具、替代 Agent Core 规划、隐藏策略判定 |
| Run Board / Task Card | 展示 Run 状态、步骤进度、阻塞原因、待审批、待接管、产物和可控操作 | 自行判断 Run 终态、在 UI 本地伪造执行进度 |
| Approval Center | 展示待处理确认、管理员审批提示、风险说明和历史决策摘要 | 绕过 Policy 决策、扩大授权范围、替代审计记录 |
| Human Handoff Console | 将 Browser/File/表单等接管对象交给用户，展示边界和接管结果回填入口 | 直接持有 Worker 内部状态、让用户在无审计边界下操作 |
| Notification Center | 汇总完成、失败、审批、接管、定时任务、后台运行和健康提醒 | 充当审批事实源、以通知替代 Run 状态 |
| Execution History / Timeline | 查看 Run Timeline、关键事件、工具摘要、截图/产物、审批和接管记录 | 展示超出权限或保留策略允许的敏感细节 |
| Settings / Preferences | 配置通知偏好、授权目录、个人记忆可见性、截图/日志偏好和低风险范围授权入口 | 修改企业硬性策略、创建未受策略约束的本地例外 |
| Local Runtime/Daemon | 提供 Run 快照、事件、审批、通知、产物和接管状态；接收控制命令 | UI 展示与用户信息架构 |
| Enterprise Control Plane | 提供策略、角色、工具/Skill、企业知识、审计查询和保留规则 | 直接读取用户本地文件或接管本地浏览器会话 |

### 3.2 事实源划分

- **执行事实源**：Local Runtime 的 Run Journal / Event Log。Desktop 只根据事件和快照渲染任务状态。
- **审批事实源**：Runtime + Policy/Approval Engine。Desktop 展示审批对象并提交用户决策，但不自行决定策略是否允许。
- **接管事实源**：Runtime 的 `handoff` 状态与 Worker Session 记录。Desktop 提供接管界面，接管开始、结束、放弃和结果摘要都由 Runtime 记录。
- **通知事实源**：Runtime 产生通知事件，Desktop 管理展示、已读、提醒偏好和历史入口。
- **治理事实源**：Enterprise Control Plane 的策略、角色、工具/Skill、企业知识库 ACL、审计查询权限和保留规则。
- **用户可见事实源**：Desktop 的 View Model，它是 Run/Approval/Handoff/Notification/Artifact/Citation 快照的投影，不保存长期真相。

## 4. 信息架构总览

Desktop Cockpit 应围绕“当前任务、待我处理、历史记录、能力与设置”组织信息，而不是按内部模块暴露 Runtime 细节。

### 4.1 一级导航

| 一级区 | 用户问题 | 主要内容 | MVP 口径 |
| --- | --- | --- | --- |
| 首页 / Agent Workspace | “我要让 Agent 做什么？” | 对话输入、快捷任务、最近 Run、当前阻塞、结果草稿 | 必须包含 |
| 任务工作台 / Run Board | “哪些任务正在运行或等待我处理？” | Run 卡片、状态过滤、待审批、待接管、后台任务 | 必须包含 |
| 审批与接管 / Decisions | “哪些动作需要我确认或接管？” | Approval Prompt、Handoff Console、风险说明、决策历史 | 必须包含，MVP 可合并在任务工作台中 |
| 执行记录 / History | “Agent 做过什么、引用了什么、生成了什么？” | Run Timeline、工具摘要、截图/产物、审批/接管记录 | 必须包含基础版 |
| 通知中心 / Notifications | “有哪些完成、失败、提醒和阻塞？” | Toast 历史、定时任务提醒、审批提醒、Runtime 健康提醒 | 必须包含基础版 |
| 工具与 Skill / Capabilities | “Agent 可以使用哪些能力？” | 工具可用性、风险等级、授权状态、Skill/模板候选入口 | MVP 做只读/候选入口 |
| 设置 / Settings | “我如何控制授权、通知、记忆和保留偏好？” | 授权目录、通知偏好、个人记忆、截图/日志偏好、范围授权 | 必须包含基础版 |

### 4.2 全局状态层

Desktop 需要在所有页面提供统一状态提示，避免用户只在某个页面才能发现风险或阻塞。

| 状态层 | 触发来源 | 展示原则 |
| --- | --- | --- |
| Runtime 连接状态 | Runtime Health Snapshot | 展示可用、重连中、恢复中、受限运行；不把 UI 断开误报为任务失败 |
| 待处理决策数 | Pending Approval / Handoff Snapshot | 明确区分“待确认”和“待接管”，提供一键进入处理入口 |
| 后台运行提示 | Run Snapshot / Scheduler Event | 告诉用户哪些任务仍在后台运行、是否安全继续、如何暂停或取消 |
| 策略限制提示 | Policy Decision Event | 说明动作被限制、需审批或被拒绝的原因；避免只显示“失败” |
| 产物完成提示 | Artifact Event | 展示报告、截图、下载文件、知识库候选写入等产物摘要和保留提示 |

### 4.3 Run 卡片信息层级

Run 卡片是任务工作台的核心视图。卡片应优先回答：目标是什么、现在处于什么状态、为什么卡住、用户能做什么、结果在哪里。

| 信息层 | 内容 |
| --- | --- |
| 任务摘要 | 用户目标、来源（对话、Scheduler、Skill、模板）、发起时间、当前状态 |
| 当前状态 | `planning`、`running`、`awaiting_approval`、`handoff`、`paused`、`recovering`、终态等用户可读说明 |
| 当前步骤 | 最近执行步骤、责任组件、进度摘要、等待原因 |
| 待处理动作 | 审批、接管、补充信息、恢复选择、取消确认 |
| 风险/策略提示 | 风险等级、策略原因、授权范围、是否可撤销 |
| 产物与引用 | 报告草稿、截图、下载文件、知识来源、文件保存位置摘要 |
| 控制操作 | 暂停、恢复、取消、查看详情、处理审批、进入接管、打开产物 |
| 审计摘要 | 关键工具调用、页面/文件访问、用户确认、错误和完成原因 |

## 5. 审批信息架构

审批提示必须把“允许什么”说清楚，而不是只问“是否继续”。审批是 Run 状态的一部分，关闭弹窗不应消除待审批对象。

### 5.1 审批提示内容

| 信息块 | 说明 |
| --- | --- |
| 动作摘要 | Agent 准备执行的动作，例如保存报告、上传文件、点击提交、修改业务状态 |
| 目标对象 | 文件路径范围、内网页面、业务记录、知识库、工具或 Skill |
| 影响范围 | 只读、写入、提交、状态变更、删除、批量操作、生产影响等 |
| 风险等级 | L0-L5 或策略细分风险，并说明为什么属于该等级 |
| 策略原因 | 企业策略、角色权限、工具声明、当前任务上下文、本次授权或本地安全兜底 |
| 可撤销性 | 可撤销、部分可撤销、不可撤销；文件操作应说明回收站/隔离区/恢复能力 |
| 证据材料 | 动作前截图、文件清单、目标页面摘要、工具参数摘要、知识引用摘要 |
| 决策选项 | 允许本次、拒绝、要求接管、取消任务；低风险范围内可提供“在此范围内始终允许” |
| 有效范围 | 当前 Run、当前步骤、当前目录、当前域名、当前工具、有限期限等 |
| 审计提示 | 决策将绑定 Run、Step、Tool Action、Policy Snapshot 和审计记录 |

### 5.2 决策选项语义

| 选项 | 适用场景 | 结果语义 |
| --- | --- | --- |
| 允许本次 | 单个 L2/L3 动作或明确 L4 动作 | Runtime 记录 Approval Event，并继续对应动作 |
| 拒绝 | 用户不希望执行该动作 | Runtime 跳过、重规划或在目标无法完成时失败 |
| 要求接管 | 用户希望亲自完成登录、验证码、提交或高风险变更 | Run 进入 `handoff`，Desktop 打开接管界面 |
| 取消任务 | 用户不再需要当前 Run | Runtime 取消 Run 并向下传播取消 |
| 范围内始终允许 | 企业策略允许的低风险、可撤销、范围明确动作 | 创建有限范围授权；策略变化或过期后必须复核 |
| 查看详情 | 用户需要更多证据再决策 | 展开 Timeline、截图、文件清单、策略原因和引用来源 |

### 5.3 审批列表与历史

- 待审批列表按紧急程度、风险等级、触发时间和是否阻塞当前用户任务排序。
- 同一 Run 的多个审批应合并展示上下文，但每个高风险动作仍需独立表达目标和影响。
- 历史记录应展示决策主体、时间、选择、范围、策略来源和后续结果摘要。
- 被拒绝或超时的审批不能从历史中消失；它们解释了 Run 为什么重规划、失败或取消。
- 管理员审批与用户确认应在文案上区分，避免用户误以为自己可以越过企业策略。

## 6. 人工接管信息架构

人工接管用于登录、MFA、验证码、文件选择、表单提交、L4/L5 业务变更、敏感操作或用户主动要求亲自操作的场景。接管不是“离开系统自己操作”，而是 Run 的 `handoff` 状态。

### 6.1 接管入口

| 入口 | 触发场景 | 展示内容 |
| --- | --- | --- |
| 审批提示中的“要求接管” | 用户不想让 Agent 自动执行动作 | 进入对应 Run 和对象的接管界面 |
| Run 卡片待处理区 | Runtime 已进入 `handoff` | 接管对象、剩余时间、风险说明、继续/放弃入口 |
| 通知提醒 | 后台任务需要 MFA、验证码或敏感操作 | 跳转到对应 Handoff Console |
| Browser Worker 页面 | 浏览器会话需要用户输入或确认 | 当前页面、允许动作边界、接管前状态摘要 |
| 文件/上传选择 | 用户需要选择本地文件或确认上传材料 | 允许目录、文件摘要、上传目标和审计提示 |

### 6.2 接管界面信息

| 信息块 | 说明 |
| --- | --- |
| 接管原因 | MFA、验证码、登录、提交确认、业务状态变更、文件选择、敏感操作或用户主动接管 |
| 接管对象 | 浏览器页面、表单、文件选择器、下载/上传、业务记录或工具动作 |
| 允许边界 | 用户可操作范围、禁止操作提示、是否允许离开当前页面或目录 |
| 风险提示 | 风险等级、不可撤销性、数据影响、是否会产生业务结果 |
| 接管前状态 | 页面标题/URL 摘要、截图引用、表单/文件清单摘要、Agent 已完成步骤 |
| 用户操作提示 | 用户需要完成什么、完成后如何交还控制、可否取消 |
| 完成回填 | 用户选择“已完成”“未完成”“放弃”“需要 Agent 重新检查” |
| 接管后摘要 | Runtime 重新读取页面/文件状态后的结果、差异和下一步 |
| 审计提示 | 接管开始、结束、用户主体、目标、结果摘要会记录到 Run Timeline |

### 6.3 接管原则

1. **边界先展示**：用户进入接管前必须知道允许操作范围和风险。
2. **用户动作与 Agent 动作分离**：审计中要区分接管期间用户完成的动作和 Agent 自动执行的动作。
3. **接管后重新观察**：Agent Core 只能基于 Runtime 重新获取的结构化状态继续，不能假设用户完成了某个业务动作。
4. **敏感输入不回显**：密码、验证码、Token、Cookie 等敏感内容不进入普通对话上下文或审计明文。
5. **放弃可恢复**：用户放弃接管时，Run 应回到可解释状态：继续等待、重规划、暂停、失败或取消。
6. **UI 关闭不丢失接管状态**：重新打开 Desktop 后必须恢复接管对象和边界，或说明 Worker 会话已失效。

## 7. 通知信息架构

通知用于把 Runtime 事件推到用户视野，但通知不是审批或任务状态的事实源。

### 7.1 通知类型

| 类型 | 来源 | 用户操作 |
| --- | --- | --- |
| 任务完成 | Run Lifecycle Event / Artifact Event | 查看结果、打开产物、查看 Timeline |
| 任务失败 | Run Lifecycle Event / Runtime Health Event | 查看原因、重试、打开记录、联系管理员 |
| 待审批 | Policy Decision Event / Approval Snapshot | 处理审批、查看风险、取消任务 |
| 待接管 | Handoff Event | 进入接管、放弃、暂停任务 |
| 定时任务提醒 | Scheduler Event | 查看草稿、暂停计划、立即处理、修改通知偏好 |
| 后台运行 | Run Snapshot / Scheduler Event | 查看运行任务、暂停、取消 |
| Runtime 健康 | Runtime Health Event | 查看受影响任务、等待恢复、导出诊断摘要 |
| 策略变化 | Policy Decision Event / Control Plane Sync | 查看受影响任务、重新授权、理解限制原因 |

### 7.2 通知展示原则

- Toast 只展示摘要和入口；详情回到 Run、Approval、Handoff 或 History 页面。
- 高风险通知不得只给“同意”快捷按钮；必须进入完整审批上下文。
- 通知历史保留完成、失败、审批、接管、调度和健康事件摘要，并遵守保留策略。
- 用户可配置通知强度，但不能关闭企业策略要求必须提醒的高风险审批或安全事件。
- 多个重复通知应合并，避免定时任务或失败重试刷屏。
- 通知必须显示来源 Run 或 Scheduler Plan，避免用户不知道提醒来自哪个任务。

## 8. 执行记录与回放信息架构

执行记录用于解释 Agent 做过什么、访问过什么、引用了什么、用户确认过什么，以及为什么成功、失败或取消。MVP 不要求完整 Replay Viewer，但事件模型和基础 Timeline 必须存在。

### 8.1 Timeline 内容

| 时间线项 | 展示摘要 |
| --- | --- |
| Run 状态变化 | 创建、规划、运行、待审批、接管、暂停、恢复、完成、失败、取消 |
| Step 事件 | 当前步骤目标、开始/完成/失败/跳过原因 |
| 策略判定 | Allow/Ask/Deny/Limit、风险等级、策略来源、范围 |
| 审批决策 | 决策人、时间、选项、范围、后续结果 |
| 接管事件 | 接管开始/结束/放弃、对象、边界、结果摘要 |
| 工具动作 | 工具类型、目标资源、执行结果、错误摘要 |
| 浏览器/文件访问 | 页面/文件范围、截图或文件清单引用、脱敏状态 |
| 知识引用 | 个人知识、企业知识、文件/网页引用、版本/过期提示 |
| 产物生成 | 报告、截图、下载文件、转换文档、任务快照 |
| 错误与恢复 | 错误类型、影响范围、恢复动作、用户可选下一步 |

### 8.2 用户可见与管理员可见

- 普通用户看到与自己 Run 相关的执行记录、产物、审批和接管摘要。
- 管理员审计查询受企业策略约束，可以按用户、工具、风险、知识库、Skill、高风险操作和错误检索，但不默认获得敏感正文。
- 敏感截图、文件内容、凭证相关信息和模型上下文必须按策略脱敏、隐藏或仅显示摘要。
- 用户删除个人产物或关闭某类记录后，后续展示应遵守新偏好；历史审计按企业保留规则处理。

## 10. 关键流程

### 10.1 交互式任务启动与观察

1. 用户在 Desktop 首页或任务工作区输入目标，选择输出形式、授权材料或已有模板。
2. Desktop 将用户目标和当前会话上下文交给 Runtime，并立即展示“已创建/正在规划”的 Run 卡片。
3. Runtime 进入 `planning` 和 `running` 后持续产生事件；Desktop 根据事件更新步骤、引用、产物和风险提示。
4. 若任务需要用户确认或接管，Run 卡片进入明显的待处理状态，并在全局 Attention Queue 和通知中心出现。
5. 完成后 Desktop 展示结果草稿、引用来源、产物、执行摘要和可追溯 Timeline。

### 10.2 高风险动作审批

1. Tool Action 被 Policy 判定为 Ask，例如文件写入、浏览器交互、提交表单、修改业务状态或知识库写入。
2. Runtime 将 Run 转入 `awaiting_approval`，并提供 Pending Approval Snapshot。
3. Desktop 展示 Approval Prompt：动作、目标、范围、风险、策略原因、可撤销性和证据材料。
4. 用户选择允许、拒绝、接管、取消或查看详情。
5. Runtime 记录 Approval Event，并继续执行、重规划、进入接管、失败或取消。
6. Desktop 在 Run Timeline 中展示决策与后续结果，通知历史标记为已处理。

### 10.3 浏览器人工接管

1. Runtime 因 MFA、验证码、登录、表单提交或 L4/L5 业务变更进入 `handoff`。
2. Desktop 打开 Handoff Console，展示接管原因、页面/对象、允许边界、风险提示和接管前状态。
3. 用户完成必要操作或选择放弃。
4. Runtime 重新获取页面/文件/业务状态，记录 Handoff Event 和接管后 Observation。
5. Agent Core 基于新 Observation 继续、重规划、完成、失败或等待下一次确认。

### 10.4 定时任务后台提醒

1. Scheduler 触发查询、汇总、草稿或提醒类 Run。
2. Desktop 不在前台时，Runtime 仍能执行安全允许的步骤并产生通知。
3. 如任务需要审批或接管，通知显示“需要处理”，但用户必须进入完整上下文再决策。
4. 用户打开 Desktop 后，首页和任务工作台展示后台 Run 的当前状态、产物和待处理项。
5. 用户可查看草稿、暂停计划、取消本次 Run 或调整通知偏好。

### 10.5 UI 关闭后的恢复

1. Desktop 关闭、崩溃或重启时，Runtime 不把 UI 连接状态当作 Run 终态。
2. 重新打开 Desktop 后，Cockpit 请求 Runtime Health Snapshot、Active Runs、Attention Queue、通知摘要和最近 Timeline。
3. Desktop 根据快照恢复任务卡、待审批、待接管、通知和产物入口。
4. 已过期的审批或失效的接管会话必须显示清楚原因，并引导用户重试、暂停、取消或等待 Runtime 重新规划。

### 10.6 执行记录查看

1. 用户从 Run 卡片、结果页、通知或历史页进入 Timeline。
2. Desktop 展示状态变化、关键步骤、策略判定、审批、接管、工具动作、产物和错误摘要。
3. 用户可打开允许范围内的截图、报告、下载文件、知识引用或错误详情。
4. 敏感内容按策略脱敏或隐藏，并说明为什么不可见。
5. 管理员审计查询由治理面完成，Desktop 只展示当前用户有权查看的记录。

## 11. 策略、审计与安全考虑

### 11.1 策略先于交互

- Desktop 展示的可用操作必须来自 Runtime/Policy 的当前状态，不得在 UI 中硬编码越权入口。
- “始终允许”只能出现在企业策略允许的低风险范围内，并且必须显示范围、期限和撤销方式。
- 策略变化后，Desktop 必须刷新受影响 Run、审批、接管和设置项，不能沿用旧授权文案。
- Scheduler 和后台通知不能因为用户不在前台而提升权限；高风险动作必须等待确认或接管。

### 11.2 审计伴随用户决策

Desktop 需要让用户理解哪些行为会进入审计：

- Run 发起、暂停、恢复、取消、重试和结果查看。
- 用户审批允许、拒绝、接管、取消和范围授权。
- 人工接管开始、完成、放弃、接管对象和结果摘要。
- 文件清单确认、截图查看、产物打开、个人知识库写入确认。
- 设置变更，例如通知偏好、授权目录、个人记忆可见性、截图/日志偏好。

审计展示必须最小化：不默认显示完整敏感页面、凭证、Cookie、Token、密码、验证码或无关文件内容。用户看到的是摘要、引用和策略允许的证据材料。

### 11.3 安全与隐私不变量

- Desktop 不读取用户日常浏览器密码、Cookie、历史记录，也不把它们展示为接管材料。
- 接管期间的敏感输入不进入普通对话上下文，不在通知或 Timeline 明文展示。
- 文件清理、移动、删除和批量整理必须先展示清单、风险和恢复方式。
- 浏览器页面内容作为不可信输入展示；涉及页面指令时应标记来源，避免用户误以为网页内容是系统指令。
- 通知中不得暴露敏感业务正文；敏感通知只展示摘要和安全入口。
- Desktop 设置不能放宽企业硬性策略；只能在允许范围内配置用户偏好。
- 个人记忆、个人知识库写入和企业知识库反馈必须有清晰入口、确认和撤销/治理说明。

## 12. MVP vs 目标态

### 12.1 MVP

MVP 必须让用户能安全、可控地使用 Agent 完成查询、汇总、草稿、知识问答、低风险浏览器和授权文件操作：

- 首页/任务工作区：对话输入、当前 Run 卡片、结果草稿、引用和产物入口。
- Run Board：运行中、待审批、待接管、暂停、失败、完成的基础过滤与控制。
- Approval Prompt：动作、目标、风险、策略原因、可撤销性、证据材料和基本决策选项。
- Handoff Console：浏览器登录、MFA/验证码、提交前接管和敏感操作接管的基础体验。
- Notification Center：完成、失败、待审批、待接管、定时任务和 Runtime 健康的基础通知历史。
- Execution History：基础 Timeline，覆盖状态、步骤、工具摘要、审批、接管、截图/产物和错误摘要。
- Settings：通知偏好、授权目录入口、个人记忆可见性、截图/日志偏好和低风险范围授权查看/撤销。
- 工具/Skill 页面：只读能力清单、风险等级和个人流程模板候选入口；不做完整 Skill Studio。

### 12.2 目标态

目标态在不改变 Runtime/Policy/Audit 语义的基础上扩展：

- 高级 Run Board：复杂过滤、批量处理、跨日期/项目/系统视图、任务依赖和健康洞察。
- 完整 Replay Viewer：可按时间回看浏览器截图、文件变更、工具观察、审批和接管边界。
- 策略生效预览：用户或管理员在执行前看到某个任务会触发哪些审批、限制和审计。
- Skill/Workflow 管理：模板候选编辑、回归测试结果、发布审核状态、版本差异和权限声明展示。
- 多通知渠道：企业 IM、邮件、移动端或系统级通知，但仍回到 Desktop/Runtime 的完整上下文处理。
- 管理员视角增强：受权限控制的审计检索、风险聚合、策略影响分析和 Runtime 健康概览。
- 可访问性与国际化增强：键盘优先审批、屏幕阅读优化、多语言风险文案和审计解释。

## 13. 设计验收清单

### 13.1 覆盖性

- [ ] Desktop 能承载对话、任务工作台、审批、接管、通知、执行记录、工具/Skill 入口和设置。
- [ ] 用户能从任意入口理解当前 Run 的目标、状态、阻塞原因、风险和可用操作。
- [ ] 待审批、待接管、后台运行和失败恢复都能进入全局 Attention Queue。
- [ ] 执行记录能展示状态、步骤、工具、审批、接管、引用、产物和错误摘要。

### 13.2 边界正确性

- [ ] Desktop 不直接执行高风险工具，不保存长期执行事实，不绕过 Runtime 改状态。
- [ ] 审批是 Runtime/Policy 状态，不是一次性 UI 弹窗。
- [ ] 人工接管是 Run 的 `handoff` 状态，接管开始/结束/放弃可审计。
- [ ] 通知不是事实源；详情必须回到 Run、Approval、Handoff 或 History 上下文。

### 13.3 治理与安全

- [ ] 高风险动作展示目标、范围、风险等级、策略原因、可撤销性和证据材料。
- [ ] “始终允许”只在策略允许的低风险、范围明确、可撤销场景出现。
- [ ] 接管期间敏感输入不进入普通对话、通知或审计明文。
- [ ] 文件清理、删除、移动、上传和知识库写入都有清单/摘要与用户确认。
- [ ] 截图、产物、日志和通知遵守敏感标记、脱敏、保留和可见范围。

### 13.4 MVP 纪律

- [ ] MVP 包含对话、Run 卡、基础审批、基础接管、通知历史、执行记录和基础设置。
- [ ] MVP 不引入完整高保真原型、复杂 Skill Studio、广泛连接器市场、高风险无人值守自动化或 Fleet 管理。
- [ ] 本设计没有 DB schema、端点级 API、代码 POC、排期估算或 SaaS 多租户扩展。

### 13.5 场景验证

- [ ] 交互式报告生成：用户能看到规划、检索、引用、草稿、产物和完成摘要。
- [ ] 受控浏览器变更：用户能在 L4/L5 动作前看到风险、截图/页面摘要，并选择确认或接管。
- [ ] 定时缺陷汇总：后台 Run 能通过通知和任务卡展示草稿、失败和待处理项，不依赖 UI 常驻。
- [ ] Skill 演进候选：重复任务可展示个人模板候选，但企业发布仍通过治理入口。
- [ ] UI 重启恢复：重新打开 Desktop 后能恢复当前 Run、待审批、待接管、通知和产物入口。

## 14. 开放问题

这些问题不阻塞本设计，但会影响后续详细设计参数：

1. MVP 首个桌面平台的系统通知能力边界是什么？是否需要托盘常驻、开机自启和锁屏通知？
2. Browser Worker 接管界面采用嵌入式受控浏览器、独立窗口还是系统浏览器式外观？不同方案会影响边界提示和截图策略。
3. 高风险审批文案需要由安全/法务统一维护模板，还是由 Policy Engine 按策略原因动态生成？
4. 通知历史与 Run Timeline 的保留周期、用户清除能力和企业审计保留如何平衡？
5. 用户是否可以为低风险动作创建个人范围授权？若可以，哪些范围维度首批开放：目录、域名、工具、任务类型还是时间？
6. 多个待审批动作是否允许批量处理？如果允许，风险分级、证据查看和审计绑定如何避免被弱化？
7. 接管期间用户如果完成了额外业务动作，Runtime 如何表达“超出 Agent 原计划但由用户完成”的结果摘要？
8. 管理员审批与普通用户确认在 Desktop 中是否共用同一 Decision Center，还是分离到管理端？
9. 个人记忆、个人知识库和企业知识库引用在结果区的视觉区分规则如何定义？
10. 审批/接管界面的无障碍要求、键盘操作和屏幕阅读文案是否需要进入 MVP 验收门槛？
