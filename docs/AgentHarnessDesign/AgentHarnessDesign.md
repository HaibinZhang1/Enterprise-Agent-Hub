# 客户端 Agent Harness 需求文档

## 1. 项目背景

当前项目需要新增一个独立的 **Agent 功能模块**，作为运行在客户端的通用型企业内网 Agent。

该 Agent 面向企业员工日常办公场景，运行在企业内网环境中，通常无法访问互联网，但可以访问企业内部系统、内网页面、内网接口和本地文件。Agent 需要具备对话、工具调用、内网页面操作、文件处理、定时任务、执行记录和权限控制能力。

本需求设计的是 **客户端 Agent Harness**，不是服务端 Agent 平台。它应作为项目中的独立功能模块存在，入口位于客户端主页，暂不与其他业务模块深度交互。

---

## 2. 项目目标

### 2.1 核心目标

构建一个运行在客户端的 Agent Harness，使用户可以通过自然语言让 Agent 协助完成企业内网环境中的日常任务。

最终效果：

```text
用户打开客户端
  ↓
从主页进入 Agent
  ↓
配置内网模型 API
  ↓
与 Agent 对话
  ↓
Agent 可访问内网链接、操作内网页面、读取/生成文件
  ↓
用户可创建定时任务
  ↓
Agent 可定时执行任务并记录过程
```

---

## 3. 设计原则

### 3.1 客户端优先

Agent Harness 运行在客户端本地，不依赖公网服务。

允许访问：

```text
1. 企业内网模型 API
2. 企业内网页面
3. 企业内网接口
4. 用户授权的本地目录
5. 本地浏览器自动化环境
```

不默认访问：

```text
1. 公网互联网
2. 未授权的本地目录
3. 其他业务模块内部数据
4. 高风险系统操作
5. 系统命令执行环境
```

---

### 3.2 功能独立

Agent 功能作为独立模块存在，第一阶段不与其他模块强耦合。

```text
允许：
- 首页提供 Agent 入口
- 设置页配置 Agent API
- 独立展示 Agent 执行记录
- 独立管理定时任务

暂不允许：
- 自动读取其他模块数据
- 自动修改其他模块业务状态
- 复用其他模块权限体系的复杂规则
- 与 Skill 市场深度集成
```

---

### 3.3 安全默认

Agent 默认只允许低风险操作。涉及写入、提交、删除、审批、支付、执行命令等动作时，必须经过权限控制或用户确认。

---

### 3.4 离线可用

企业内网环境中，Agent Harness 不应依赖公网安装、联网拉取依赖或访问云端服务。

模型、工具、浏览器自动化能力、任务调度能力都应支持内网离线运行。

---

## 4. 范围说明

## 4.1 本期范围

本期建设内容：

```text
1. Agent 首页入口
2. 模型 API 配置
3. Agent 对话与任务执行
4. 工具调用机制
5. 内网页面自动化
6. 本地文件读写
7. 定时任务
8. 用户确认机制
9. 执行记录
10. 基础权限控制
```

---

## 4.2 暂不建设范围

以下能力暂不纳入第一阶段：

```text
1. 多 Agent 协作
2. 服务端 Agent 平台
3. 企业级统一 Agent 编排中心
4. 自动扫描用户电脑文件
5. 自动索引企业知识库
6. 与现有业务模块深度集成
7. 完整 Skill 市场
8. 任意 Shell 命令自动执行
9. 高风险业务自动提交
10. 跨用户共享记忆
11. 云端同步
12. 复杂工作流设计器
```

---

# 5. 优先级定义

## P0：必须实现

P0 是 Agent Harness 能成立的最小闭环。没有 P0，Agent 功能不可上线。

目标是实现：

```text
能配置模型
能打开 Agent
能对话
能调用基础工具
能访问内网页面
能读写授权文件
能创建定时任务
能进行用户确认
能记录执行过程
```

---

## P1：增强体验

P1 是提升可用性、稳定性和企业办公体验的能力。P0 上线后应优先补齐。

---

## P2：扩展能力

P2 是面向后续平台化、技能化、企业级治理的扩展能力。

---

## P3：长期规划

P3 是未来可考虑的高级能力，不影响当前 MVP 上线。

---

# 6. P0 需求

# P0-1：首页 Agent 入口

## 需求说明

客户端主页需要提供独立的 Agent 功能入口。

入口名称建议：

```text
AI 助手
Agent 助手
企业助手
智能任务助手
```

首页入口点击后进入 Agent 模块主页。

---

## 功能要求

Agent 模块主页至少包含：

```text
1. 开始对话
2. 定时任务
3. 工具能力
4. 执行记录
5. Agent 设置
```

---

## 页面结构建议

```text
Agent 首页
├─ 对话助手
│  └─ 进入 Agent 对话窗口
├─ 定时任务
│  └─ 查看和创建周期任务
├─ 工具能力
│  └─ 查看当前 Agent 可用工具
├─ 执行记录
│  └─ 查看历史任务、失败原因、执行过程
└─ 设置
   └─ 配置模型 API、权限、日志等
```

---

## 验收标准

```text
1. 用户可以从客户端主页进入 Agent 模块
2. Agent 模块有独立首页
3. 首页可以进入对话、定时任务、工具能力、执行记录和设置
4. 关闭 Agent 页面不影响整个客户端运行
```

---

# P0-2：模型 API 配置

## 需求说明

Agent Harness 必须支持用户配置企业内网模型 API。

第一阶段优先支持 OpenAI-Compatible API。

---

## 配置项

```text
1. API Base URL
2. API Key
3. Model Name
4. Timeout
5. Max Tokens
6. Temperature
7. 是否启用流式输出
8. 是否启用工具调用
```

---

## 功能要求

```text
1. 用户可以新增、编辑、删除模型配置
2. 用户可以设置默认模型
3. 用户可以测试 API 连通性
4. 测试失败时需要展示明确错误
5. API Key 需要本地安全存储
6. Agent Core 不应直接绑定某一家模型厂商 SDK
```

---

## 模型能力要求

配置页需要展示或允许配置模型能力：

```text
1. 是否支持工具调用
2. 是否支持 JSON 输出
3. 是否支持流式响应
4. 是否支持视觉能力
5. 上下文长度
```

如果模型不支持原生 tool calling，应允许降级为 JSON 工具调用协议。

---

## 验收标准

```text
1. 用户可以配置一个内网模型 API
2. 可以测试连接是否成功
3. 测试成功后可以在 Agent 对话中使用
4. 模型请求失败时，前端能展示可理解的错误信息
5. API Key 不以明文展示
```

---

# P0-3：Agent 对话与任务执行

## 需求说明

Agent 模块需要提供基础对话能力，并能够围绕用户任务进行多步执行。

---

## 功能要求

```text
1. 用户可以输入自然语言任务
2. Agent 可以回复自然语言
3. Agent 可以根据任务选择工具
4. Agent 可以多轮执行工具调用
5. Agent 可以根据工具返回结果继续推理
6. Agent 可以输出最终结果
7. 用户可以中断当前执行
```

---

## Agent 执行流程

```text
用户输入任务
  ↓
Agent 创建任务运行实例
  ↓
构造上下文
  ↓
调用模型
  ↓
模型返回回复或工具调用
  ↓
工具调用经过权限判断
  ↓
Runtime 执行动作
  ↓
返回观察结果
  ↓
Agent 继续判断下一步
  ↓
输出最终结果
```

---

## 执行状态

P0 至少支持以下状态：

```text
1. 等待执行
2. 正在思考
3. 正在调用工具
4. 等待用户确认
5. 已完成
6. 已失败
7. 已取消
```

---

## 验收标准

```text
1. 用户可以正常与 Agent 对话
2. Agent 可以根据任务调用至少一种工具
3. 工具调用结果可以返回给 Agent 继续处理
4. 用户可以取消正在执行的任务
5. 失败时能展示失败原因
```

---

```text
1. browser.*
2. http.*
3. file.*
4. notification.*
5. scheduler.*
```

## P0 内置工具

### browser 工具

```text
browser.open
browser.extract_text
browser.screenshot
browser.click
browser.fill
browser.wait_for
```

### http 工具

```text
http.get
http.download
```

### file 工具


### scheduler 工具

## 暂不默认开放的工具

```text
system.shell

```

如后续开放，必须作为高风险能力，并由管理员显式启用。

---

## 工具基本信息

每个工具需要包含：

```text
1. 工具名称
2. 工具描述
3. 输入参数说明
4. 输出结果说明
5. 权限等级
6. 是否需要用户确认
7. 所属执行器

```

---

## 验收标准

```text
1. Agent 只能调用已注册工具
2. 禁用工具不会暴露给 Agent
3. 工具调用参数需要校验
4. 工具调用失败时有统一错误返回
5. 工具列表可以在 UI 中查看
```

# P0-5：Runtime Executor 执行器

## 需求说明

Runtime Executor 负责真正执行 Agent 发起的工具调用。

Agent Core 只负责决策，不直接操作浏览器、文件或网络。

---

## 执行器类型

P0 至少包含：

```text
1. Browser Executor
2. HTTP Executor
3. File Executor
4. Notification Executor
5. Scheduler Executor
```

---

## 执行原则

```text
1. 所有工具调用必须转为标准 Action
2. 所有执行结果必须转为标准 Observation
3. 执行前需要进行参数校验
4. 执行前需要进行权限判断
5. 执行后需要记录执行结果
6. 执行失败需要返回可理解错误
```

---

## Action 内容要求

Action 至少需要包含：

```text
1. 工具名称
2. 输入参数
3. 任务 ID
4. 会话 ID
5. 权限等级
6. 是否需要确认
7. 执行超时时间
```

---

## Observation 内容要求

Observation 至少需要包含：

```text
1. 执行状态
2. 执行摘要
3. 返回数据
4. 错误信息
5. 相关截图或文件产物
6. 开始时间
7. 结束时间
```

---

## 验收标准

```text
1. Agent 工具调用通过 Runtime Executor 执行
2. Runtime Executor 能路由到不同执行器
3. 工具执行成功后返回统一格式结果
4. 工具执行失败后返回统一格式错误
5. 所有执行动作都能被记录
```

---

# P0-6：内网浏览器自动化

## 需求说明

Agent 需要能够访问企业内网页面，并协助用户完成页面阅读、点击、填写、查询等操作。

---

## 功能要求

P0 支持：

```text
1. 打开内网 URL
2. 获取页面标题
3. 提取页面文本
4. 页面截图
5. 点击按钮或链接
6. 填写输入框
7. 等待页面加载或指定元素出现
8. 失败时截图留存
9. 用户人工接管后继续执行
```

---

## 浏览器运行方式

建议使用受控浏览器环境，例如：

```text
1. Chromium
2. Playwright
3. 系统 WebView + 自动化桥接
```

优先建议 Playwright，因为后续截图、DOM 提取、点击、填写、下载等能力更完整。

---

## 页面目标定位

P0 支持以下定位方式：

```text
1. 文本匹配
2. 按钮名称
3. 输入框 label
4. CSS selector
5. 简单自然语言目标描述
```

---

## 人工接管

以下场景需要支持人工接管：

```text
1. 登录失败
2. 验证码
3. UKey
4. 页面控件无法识别
5. Agent 连续操作失败
6. 高风险操作前
7. 用户主动要求接管
```

人工接管流程：

```text
Agent 暂停
  ↓
展示当前浏览器页面
  ↓
用户手动操作
  ↓
用户点击“继续”
  ↓
Agent 重新观察页面
  ↓
继续执行
```

---

## 验收标准

```text
1. Agent 可以打开内网 URL
2. Agent 可以读取页面文本
3. Agent 可以保存页面截图
4. Agent 可以点击页面按钮或链接
5. Agent 可以填写表单字段
6. 遇到复杂页面时可以交给用户接管
7. 用户接管后 Agent 可以继续执行
```

# P0-7：本地文件能力

## 需求说明

Agent 需要具备基础文件处理能力，但必须限制在用户授权目录内。

---

## 功能要求

P0 支持：

```text
1. 读取授权目录内文件
2. 写入授权目录内文件
3. 列出授权目录内文件
4. 生成文本文件
5. 保存下载文件
6. 保存 Agent 生成的报告
```

---

## 文件访问限制

```text
1. 默认不能访问全盘文件
2. 用户需要选择授权目录
3. Agent 只能访问授权目录及其子目录
4. 敏感目录默认禁止访问
5. 删除文件能力 P0 不开放
```

---

## 支持文件类型

P0 优先支持：

```text
1. txt
2. md
3. json
4. csv
5. log
6. html
7. 常见下载文件的保存
```

复杂 Office 文档解析可放到 P1 或 P2。

---

## 验收标准

```text
1. 用户可以配置 Agent 授权目录
2. Agent 可以读取授权目录内文件
3. Agent 可以写入授权目录内文件
4. Agent 无法访问未授权目录
5. 文件读写行为有执行记录
```

---

# P0-8：定时任务

## 需求说明

Agent 需要支持定时执行任务。

定时任务是本项目的重要目标，因此 P0 必须具备基础能力。

---

## 功能要求

P0 支持：

```text
1. 创建一次性任务
2. 创建周期任务
3. 暂停任务
4. 恢复任务
5. 删除任务
6. 手动立即执行任务
7. 查看任务执行历史
8. 任务失败后通知用户
```

---

## 定时方式

P0 支持：

```text
1. 指定时间执行一次
2. 每天固定时间执行
3. 每周固定时间执行
4. 每月固定日期执行
```

Cron 表达式可作为高级模式放到 P1。

---

## 任务内容

定时任务本质上是一个 Agent Prompt。

例如：

```text
每天上午 9 点打开 OA 公告页面，检查是否有新通知，如果有则提醒我。
```

```text
每周五下午 5 点打开报表系统，下载本周报表并保存到指定目录。
```

---

## 执行原则

```text
1. 每次定时触发都创建独立执行记录
2. 同一个任务不允许并发执行
3. 上一次未完成时，本次应跳过或排队
4. 高风险操作不能因为是定时任务就自动执行
5. 失败时需要通知用户
6. 用户可以随时停用任务
```

---

## 后台运行

客户端需要支持后台常驻能力。

建议拆分为：

```text
1. UI 主程序
2. Agent Daemon / Sidecar
```

UI 关闭后，是否继续运行定时任务应由用户配置。

---

## 验收标准

```text
1. 用户可以创建一个每天执行的 Agent 任务
2. 到达时间后任务可以自动触发
3. 任务执行过程有记录
4. 任务失败后用户能收到通知
5. 用户可以暂停、恢复、删除任务
6. 同一任务不会重复并发执行
```

---

# P0-9：权限与用户确认

## 需求说明

Agent 需要具备基础权限控制和用户确认机制。

---

## 权限等级

P0 使用四级权限：

```text
L0_READ
只读操作，例如读取页面、截图、读取授权文件。

L1_LOCAL_WRITE
本地写入操作，例如生成文件、保存报告、保存下载内容。

L2_EXTERNAL_WRITE
外部系统写入操作，例如提交表单、上传文件、调用 POST 接口。

L3_DANGEROUS
高风险操作，例如删除、审批、转账、支付、执行命令。
```

---

## 默认策略

```text
L0_READ：允许自动执行
L1_LOCAL_WRITE：允许自动执行，可配置确认
L2_EXTERNAL_WRITE：必须用户确认
L3_DANGEROUS：默认禁止
```

---

## 必须确认的操作

```text
1. 提交表单
2. 上传文件
3. 调用写入型接口
4. 点击“提交”“确认”“删除”“审批”“支付”“转账”等按钮
5. 执行可能改变系统状态的操作
```

---

## 确认弹窗内容

确认弹窗需要展示：

```text
1. 当前任务
2. 即将执行的工具
3. 操作目标
4. 页面地址或文件路径
5. 风险等级
6. Agent 执行理由
7. 允许本次
8. 拒绝本次
```

P0 不提供“永久允许所有类似操作”。

---

## 验收标准

```text
1. 只读操作可以自动执行
2. 外部写入操作会触发用户确认
3. 用户拒绝后 Agent 停止对应动作
4. 高风险操作默认无法执行
5. 每次用户确认都有记录
```

---

# P0-10：执行记录与审计

## 需求说明

Agent 的每次任务执行都必须有记录，便于用户查看过程、排查失败原因和追溯风险操作。

---

# 13. 当前建议

第一阶段不要把 Agent 做成复杂平台，而是先实现一个可靠的客户端 Agent Harness：

```text
P0 核心闭环：
模型配置
  ↓
Agent 对话
  ↓
工具注册
  ↓
Runtime 执行
  ↓
权限确认
  ↓
执行记录
  ↓
定时任务
```

优先做出一个能在内网真实跑通的版本，再逐步扩展 Skill、MCP、沙箱、企业策略和服务端审计。
