## Design Context

### Users
Enterprise Agent Hub serves mixed internal users in an enterprise desktop setting: ordinary employees discovering and installing Skills, authors publishing Skills, reviewers handling governance queues, and administrators managing departments, users, permissions, and Skill lifecycle state. They use the product as a working tool rather than a marketing surface, often with dense lists, status checks, installation/enablement decisions, and live service or local runtime feedback in view.

### Brand Personality
Technology-forward, fast, and tool-like. The interface should feel precise and efficient, closer to a refined command workspace than a decorative SaaS site. It should suggest Raycast-level immediacy and Linear-level clarity while staying simpler than visually busy admin dashboards.

### Aesthetic Direction
Use a light desktop workbench as the default: cool white surfaces, tinted neutral depth, pine/seafoam accents, sharp hierarchy, and restrained motion. Avoid noisy decoration, high-saturation gradients, cyber glow, heavy glass effects, nested cards, and overly complex visual systems. The product should feel like a high-trust internal instrument: compact, responsive, inspectable, and calm under operational pressure.

### Design Principles
1. Make the main task obvious within three seconds: primary action, current status, and selected object must be visually dominant.
2. Favor fast-tool density over landing-page drama: compact controls, strong alignment, and scannable rows should carry the experience.
3. Use color sparingly for meaning: status, focus, and primary actions get emphasis; decorative color stays quiet.
4. Keep governance flows calm: destructive, permission, offline, and update states must feel explicit without becoming visually loud.
5. Preserve Chinese product copy and desktop-first behavior while keeping narrow layouts usable instead of hiding core actions.

## Global Refactor Brief

### 1. Feature Summary
Refactor the desktop UI globally so the app reads as one cohesive, production-grade enterprise tool across home, community, local, review, and admin workbenches. The target audience is mixed internal users who need speed, clarity, and confidence while browsing, installing, publishing, reviewing, and administering Skills.

### 2. Primary User Action
At any moment, the user should quickly understand where they are, what object is selected, what state it is in, and what the next safe action is.

### 3. Design Direction
Express the brand as a fast, precise, tool-like desktop workspace. Use Raycast as an inspiration for immediacy and crisp command surfaces, Linear as inspiration for quiet hierarchy and density, and the existing requirement docs as the product guardrail: light enterprise AI workbench, dense but clear, restrained motion, no decorative complexity.

### 4. Layout Strategy
Strengthen the global shell first: topbar, navigation, workbench grids, panels, list rows, inspector panels, and shared control surfaces. Reduce visual noise by standardizing radii, spacing, borders, and elevation. Keep cards only where they represent distinct actionable objects; use spacing, dividers, and typography inside workbench panels rather than creating nested card stacks.

### 5. Key States
Default workbench state should emphasize selected context and available actions. Empty states should teach the next step. Loading and scanning states should be calm and explicit. Error, offline, permission, destructive, and update states should use semantic colors with copy-first clarity. Dense admin states should remain scannable when lists grow large.

### 6. Interaction Model
Interactions should feel immediate: buttons, rows, nav items, filters, and tabs get clear hover/focus/active feedback. Hover may enhance affordance but must not reveal critical functionality. Motion should be subtle and transform/opacity based, with reduced-motion support.

### 7. Content Requirements
Preserve existing Chinese copy unless a requirement change explicitly calls for wording changes. Labels should stay short and result-oriented. Status text should remain globally consistent with the requirement documents.

### 8. Recommended References
Use `spatial-design.md`, `typography.md`, `color-and-contrast.md`, and `responsive-design.md` for this pass. Use `interaction-design.md` and `motion-design.md` only for focused follow-up work on complex flows.

### 9. Open Questions
No blocking questions for the first pass. Keep the refactor visual and structural in `apps/desktop`, preserving existing state/data behavior and avoiding new dependencies.
